Overview

Webserv is a fully custom HTTP server built in C++98, developed as part of the 42 school curriculum. The objective was to create a functional web server capable of handling multiple simultaneous clients, parsing configuration files, and responding correctly to standard HTTP requests (GET, POST, DELETE), all without using any external libraries or frameworks. This project offers a deep dive into low-level networking, protocol design, and systems programming.

Project Objectives

  • Build a HTTP/1.1-compliant server in C++98.
  • Support multiple server blocks and routes through configuration files.
  • Implement request handling for GET, POST, DELETE methods.
  • Manage multiple client connections using non-blocking I/O and poll().
  • Handle CGI scripts securely and efficiently.
  • Ensure complete stability and robustness even under memory stress or malformed requests.

Architecture Overview

The server is designed with modularity and separation of concerns in mind. Below is a breakdown of the major components:

1. Configuration Parser

  • Tokenizer: Splits raw .conf file content into lexical tokens.
  • Parser: Converts the stream of tokens into structured server configuration objects (e.g., Config, ServerConfig, LocationConfig).
  • Supports nested blocks and inheritance of directives.

2. Server Core (HTTPServer)

  • Initializes TCP sockets and binds to configured ports.
  • Uses poll() to monitor multiple client connections without blocking.
  • Manages new connections and maps them to ClientConnection objects.

3. Request/Response Handling

  • HTTPRequest: Parses raw HTTP requests (headers, body, method, etc.).
  • HTTPResponse: Generates proper HTTP responses based on status codes, headers, and configured routes.
  • Handles method-based routing (GET, POST, DELETE) with appropriate error management.

4. CGIHandler

  • Supports execution of external scripts (e.g., .py, .pl) using forked processes.
  • Captures script output and sends it back as part of the HTTP response.
  • Ensures environment variables are passed properly for CGI compliance.

5. Error Handling & Defaults

  • Customizable error pages per server block.
  • Defaults for missing files, unsupported methods, malformed requests.

Development Process

We followed a structured and time-bound approach to project development:

Planning & Research (Week 1)

  • Read and discussed project subject.
  • Divided core responsibilities (parsing, server core, HTTP logic).
  • Reviewed key documentation (Beej's Guide, IBM docs, real open-source webserv repos).

  • Developed tokenizer and recursive-descent parser.

  • Request parsing, method routing, and dynamic response generation.

Challenges & Resolutions

Parsing Complexity

Designed a robust tokenizer to distinguish between nested blocks, directives, and symbols ({, }, ;).

Non-blocking I/O

Carefully handled edge cases with poll() such as timeouts and closed sockets.

CGI Handling

Ensured proper isolation and input/output redirection using child processes.

Memory Management

Avoided memory leaks and undefined behavior, respecting C++98 limitations.

Conclusion

Webserv was a challenging yet rewarding systems-level project that required not only deep technical skills but also strong coordination, documentation, and planning. Each team member contributed to building a fully functional web server from scratch, while gaining practical experience with sockets, HTTP internals, file parsing, and process management.

This project was a real-world simulation of building software from the ground up — starting from a blank file, defining specifications, dividing ownership, and debugging through iteration.
